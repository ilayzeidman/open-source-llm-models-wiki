## Definition

"A high-throughput, low-latency open-source inference serving framework for
deploying generative AI and reasoning models in large-scale distributed
environments." Functions as an **orchestration layer above** inference engines
(vLLM, SGLang, TensorRT-LLM), not a replacement.

## Components

- **Frontend** — OpenAI-compatible API gateway.
- **Workers** — engine instances (vLLM, SGLang, TRT-LLM).
- **Smart Router (KV-aware)** — "hashes incoming requests and stores them in a
  Radix Tree, which allows the tracking of KV locations in large scale
  distributed inference environments." Yields "2x faster time-to-first-token
  on large models like Qwen3-Coder 480B."
- **Planner / SLO Planner** — "dynamically decide whether to use disaggregated
  or aggregated serving, or to shift GPU resources between prefill and decode
  phases." Real-world: "80% fewer SLA breaches at 5% lower TCO" (Alibaba
  APSARA 2025).
- **NIXL** — "high-throughput, low-latency point-to-point communication
  library." Five backends: RDMA/InfiniBand, RoCE via UCX, TCP fallback, NVMe-oF,
  S3-compatible object storage. Transfers KV "directly from the VRAM of
  prefill engine to the VRAM of decode engine," non-blocking.
- **KV Block Manager (KVBM)** — three layers (model integration, memory mgmt,
  storage). Hierarchical offload: GPU HBM → CPU DRAM → local SSD → remote
  network storage. Vast Data measured **35 GB/s to a single H100**; WEKA
  **270 GB/s across 8 GPUs** via RDMA zero-copy.
- **Grove** — topology-aware gang scheduler for NVL72.
- **DynamoGraphDeploymentRequest (DGDR)** CRD for K8s.
- **ModelExpress** — streams weights GPU-to-GPU over NIXL/NVLink: "7x faster
  cold-start for new replicas."

## Disaggregated prefill/decode (design doc)

"The prefill and decode phases of LLM requests have different computation
characteristics and memory footprints." Three-step flow: prefill compute → KV
transfer → decode.

- Decode workers also handle short prefills (route decisions made there).
- Prefill workers "pull from prefill queue, reads KV blocks with prefix cache
  hit from the worker, computes the prefill, and writes the computed KV blocks
  back."
- Conditional disaggregation: requests go remote only when prefill length >
  threshold AND prefill queue not overloaded — "allows Dynamo to achieve high
  performance for dynamic workloads."
- Runtime elasticity: "Workers and prefill workers can be added and removed at
  runtime without any system-level synchronization or overheads."

## Performance claims

- "30× throughput" on DeepSeek-R1 on Blackwell at launch (GTC 2025).
- Dynamo 1.0 (Mar 2025): production GA.
- Dynamo 1.1.0 (May 2026): multimodal disaggregation (30% faster TTFT on
  images), S3/Azure blob KV offload.
- GTC 2026: 7× throughput improvement.
- "doubles the performance and revenue of AI factories serving Llama models on
  Hopper" (NVIDIA newsroom).
- "Boost the number of tokens generated by over 30x per GPU" on DeepSeek-R1 on
  GB200 NVL72.

### Real-hardware numbers (SemiAnalysis InferenceMAX, late 2025)

- GB200 NVL72 with Dynamo+vLLM: **12,587 tok/s/GPU** on Kimi K2.5 NVFP4 8k/1k.
- Single B200 node (no Dynamo): **4,021 tok/s/GPU**.
- ~3× per-GPU win from disaggregation at scale.

### Simulation-based numbers (NVIDIA Dynamo + GB200 NVL72 blog)

- "6x throughput performance gain… in the medium latency regime" for DeepSeek-R1.
- "up to 3x" for Llama 70B.
- Caveat: "evaluated hundreds of thousands of design points across diverse
  hardware configurations using a high-fidelity data-center scale GPU
  performance simulator" — these are simulated.

## SLA model (Dynamo Planner)

- "Time to First Token (TTFT) requirements applying to prefill"
- "Inter-Token Latency (ITL) requirements guiding decode deployment choices"
- Dynamic rate matching: "resources are allocated based on load across the
  prefill and decode phases."

## AWS EKS deployment notes

- Works with NVIDIA GPU instances "P6, P5, P4d, P4de, G5, and G6."
- Karpenter "burst new g6 instances in less than 60 seconds."
- "EFA is used to talk between the GPU nodes within a single Availability
  Zone."
- Model storage: Amazon EFS (NFS-style RWX shared by all pods).

## Single-GPU value

> Dynamo's own README is direct: **"If you're running a single model on a
> single GPU, your inference engine alone is probably sufficient."**

For a single g5.xlarge / g6e.xlarge, run vLLM (or SGLang) directly. Dynamo
earns its keep multi-GPU / multi-node.

## Operational gotcha: `/v1/models` answers empty during startup

The `dynamo.frontend` HTTP service comes up in seconds — well before any
worker has finished pulling or loading weights. Until a `dynamo.vllm` worker
registers itself in etcd, the frontend answers `GET /v1/models` with HTTP 200
and `{"object":"list","data":[]}`.

```bash
# good — only true once a worker has registered the model in etcd
curl -fsS http://localhost:8000/v1/models | grep -q '"id"'
```
